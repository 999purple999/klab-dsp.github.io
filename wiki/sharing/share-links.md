## What it is

A share link is a URL that gives anyone who holds it access to a specific note, at a specific permission level (View, Comment, or Edit), for a configurable period of time. The defining property of K-Perception share links is that the decryption key for the note content is embedded exclusively in the URL fragment — the `#…` portion of the URL — and is therefore never transmitted to the server, never written to server logs, and never visible to K-Perception or any network intermediary. The server stores only AES-256-GCM ciphertext alongside an initialisation vector (IV). Without the fragment key, that ciphertext is computationally indistinguishable from random noise.

This makes K-Perception share links structurally different from links generated by conventional cloud tools such as Google Docs or Notion, where the service provider can read the shared content at any time. K-Perception cannot read your shared notes.

## When to use it

Use a share link whenever you want to give someone access to a note without requiring them to create a K-Perception account. Common scenarios:

- Sharing a research summary with a colleague who does not use K-Perception.
- Publishing a Markdown document as a read-only web page for public reference.
- Giving a reviewer Comment-level access to a draft so they can annotate it without altering the source.
- Enabling real-time co-editing with another person via Edit-level access backed by the Y.js collaborative layer.
- Distributing a time-limited briefing that automatically expires after 12 hours (or your chosen duration).

Share links work on any device with a modern browser — recipients do not need the K-Perception app installed.

## Step by step

### Creating a share link

1. Open the note you want to share in the K-Perception editor.
2. Click the **Share** button in the toolbar (or press **Ctrl/Cmd + Shift + S**).
3. The Share dialog opens. Choose a **permission level**: View, Comment, or Edit.
4. Set an **expiry**. Local plan users are fixed at 12 hours. Paid plan users can choose from 1 hour, 6 hours, 12 hours, 1 day, 7 days, 30 days, or No expiry.
5. Optionally enable **Password protection** and enter a passphrase (see the Password Protection article for details).
6. Click **Generate link**. K-Perception derives a fresh 32-byte random key, encrypts the note content with AES-256-GCM, uploads the ciphertext and IV to the sync worker, and appends the key as the URL fragment.
7. The full URL is displayed and copied to your clipboard. Share it through any channel — email, messaging app, or QR code.

### Managing existing links

- All active share links for a note are visible in the Share dialog under **Active links**.
- Each link shows: permission level, expiry date/time, access count, and creation date.
- Click **Copy** next to any link to copy it again.
- Click **Revoke** to immediately invalidate a link (see the Revocation article for details).

### Editing a link

Share links cannot be edited after creation (permission level, expiry, and password are all baked into the encrypted token). To change any property, revoke the existing link and create a new one.

## Behaviour and edge cases

**Expired links.** Once the expiry timestamp passes, the share viewer returns a `410 Gone` response and displays an expiry message to the recipient. The ciphertext is retained in D1 for a short grace period before garbage collection.

**Concurrent active links.** You may have multiple active share links for the same note with different permission levels — for example, a public View link and a private Edit link for a specific collaborator.

**Note updates.** A share link always reflects the current state of the note at the moment the recipient opens it (the share viewer fetches the latest ciphertext). If you update the note after creating the link, recipients will see the updated content on their next page load.

**Offline creation.** Share links require a connection to the sync worker at creation time (to upload the ciphertext). You cannot create share links while offline.

**Draft notes.** Notes that have never been synced cannot be shared until they are synced at least once.

**Local plan link limit.** Local (free) accounts are limited to 2 simultaneously active share links across all notes. Attempting to create a third link on the Local plan prompts an upgrade dialog.

**Fragment handling in messaging apps.** Some messaging apps strip the `#fragment` from URLs when generating link previews. If a recipient reports the link is broken, instruct them to open the full URL in a browser rather than tapping the in-app preview.

## Permissions and roles

The permission level encoded in the share link determines what the recipient can do at the share viewer:

- **View** — read-only rendering of the note; no interaction with note content.
- **Comment** — all View capabilities plus the ability to post and reply to threaded comments.
- **Edit** — full collaborative editing via the Y.js layer; the recipient can modify the note content in real time.

Workspace role-based access control (RBAC) is separate from share link permissions. A workspace Admin cannot use a share link to bypass workspace ACL; share links apply only to the share viewer at `kperception-share.pages.dev`.

## Security implications

This section is intentionally detailed because share links involve content leaving the encrypted vault.

### The fragment-key principle

Per [RFC 3986 §3.5](https://datatracker.ietf.org/doc/html/rfc3986#section-3.5), the fragment component of a URI (`#…`) is processed exclusively by the client (browser or app). It is **not** included in HTTP request paths, **not** sent to the server in any request header or body, and **not** written to server access logs. K-Perception exploits this property deliberately: the 32-byte AES-256-GCM decryption key lives only in the fragment. Every other component of the URL (scheme, host, path, query string) is transmitted to and logged by the server, but none of those components are sufficient to decrypt the note.

### What an attacker who compromises the server gets

The server (Cloudflare Worker + D1 database + R2 storage) holds:

- AES-256-GCM ciphertext of the note content.
- A 12-byte random IV used for that encryption.
- Metadata: share token hash, permission level, expiry timestamp, access count, creation timestamp.
- Nothing else.

AES-256-GCM with a random 32-byte key provides 256-bit security. Brute-forcing the key is not computationally feasible. A server-side attacker who dumps the entire D1 database and R2 store cannot recover note content.

### What a network eavesdropper gets

HTTPS encrypts the HTTP request in transit, but even without HTTPS, the fragment is never included in the HTTP request line. A passive network observer who captures the raw TCP traffic (or an HTTP proxy that logs requests) will see the path `/share/{tokenId}` but will not see `#{keyHex}`. A full TLS intercept (e.g., corporate MITM proxy) similarly does not capture the fragment. The only vector is observing the full URL string in the browser address bar or in browser history on the recipient's device.

### What K-Perception sees

Nothing. The sync worker receives requests to the path `/share/{tokenId}`, returns the ciphertext and IV, and records an access event with a timestamp. The decryption key is never present in any request the server receives. K-Perception staff, Cloudflare staff, and any legal compulsion process directed at the server infrastructure cannot recover note content from share links.

### Threat model boundaries

The following threats are **out of scope** for the share-link design:

- **Recipient device compromise.** If the recipient's browser or OS is compromised, the attacker can read the decrypted content from memory or the browser DOM. K-Perception cannot protect against endpoint compromise.
- **URL exfiltration.** The full URL including fragment may appear in the recipient's browser history, clipboard, or message thread. Treat share link URLs as secrets equivalent to the note content itself.
- **Edit-level links.** An Edit-level recipient can copy the decrypted content from the share viewer. Granting Edit access implies trusting the recipient with the full note text.
- **Pre-revocation access.** Any content already rendered in a recipient's browser before revocation has been delivered. Revocation prevents future access, not past access.

### Key derivation

Each share link uses a freshly generated 32-byte random key (`crypto.getRandomValues`). The key is not derived from the note's vault key, the user's password, or any other secret. This means:

- Revoking a share link does not require re-encrypting the note.
- Rotating your vault key does not affect existing share links (they use independent keys).
- Compromising one share link key does not reveal the vault key or any other note.

### Password-protected links

When password protection is enabled, the 32-byte share key is wrapped with a PBKDF2-derived wrapping key (SHA-256, 200 000 iterations, random 16-byte salt). The wrapped key and salt are stored in D1. The bare share key is never stored anywhere. See the Password Protection article for the full derivation path.
