Security architecture

Zero-knowledge by design.
Not by marketing.

Every architectural decision passes one test: if the server and its databases were subpoenaed or breached, could plaintext content be recovered? The answer is no, and we can show you why.

AES-256-GCM
NIST-approved · NSA Suite B
Authenticated encryption via Galois/Counter Mode. 256-bit key, 2²⁵⁶ brute-force space. The GCM tag detects any modification: even a single bit-flip is rejected.
PBKDF2-SHA256 · 100K iter
NIST SP 800-132 compliant
Master password stretched through 100,000 PBKDF2-SHA256 rounds with a unique random 16-byte salt. Brute-forcing requires millions of operations per password attempt.
HKDF Key Hierarchy
WDK → SK_section → SK_dir → DEK
HKDF-derived per-section and per-directory keys. Per-file Data Encryption Keys mean revocation costs only one DEK rotation, not a full vault re-encrypt.
PKCE OAuth · Google ID
RFC 7636 · Proof Key for Code Exchange
Login uses OAuth 2.0 PKCE. The proof key never travels to the server in plaintext. Login is identity-only and never carries the vault decryption key.
Biometric Enclave
Android Keystore · TEE
Vault session key stored in hardware-isolated TEE via Capacitor biometric API. Inaccessible to user storage, full backups, or ADB extraction.
Share Cryptography
URL #fragment · never to server
Random 32-byte key delivered via URL fragment (RFC 3986 §3.5: never in HTTP requests). Optional PBKDF2 password gate. Server-side instant revocation via revokeTokenHash.
Elevation Tokens
TOTP step-up · server-validated
Destructive ops (rotate-key, remote-wipe, transfer-ownership, restore-backup) require time-limited tokens issued only after step-up. Cannot be bypassed via API manipulation.
Zero Server Plaintext
D1 + R2 hold ciphertext only
Server stores: blob size, user ID, timestamps, change IDs. Server cannot read: titles, bodies, tag names, filenames, search queries, Y.js document edits.
